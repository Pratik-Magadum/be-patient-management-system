## 1. Project Overview

The **Patient Management System (PMS)** backend service manages multi‑tenant hospital operations including:

| Domain Module       | Responsibility                                        |
|---------------------|-------------------------------------------------------|
| `patient`           | Patient master records (CRUD + search)                |
| `appointment`       | Patient appointment scheduling                        |
| `consultation`      | Doctor consultation & prescription records            |
| `diagnostic`        | Pre‑diagnostic test results (vision, eye pressure)    |
| `prescription`      | Medicine prescriptions linked to consultations        |

**Tenant isolation** is enforced at the application layer via `hospital_id` on every query.  
**Patient numbers** (`PT-YYYY-NNNNN`) are auto‑generated by a PostgreSQL trigger.

## 2. Architecture & Design Principles

### Layered Architecture

```
┌──────────────────────────────────┐
│   Controller Layer (REST API)    │  ← HTTP In / Response Out (DTO)
│   Interface + Implementation     │
├──────────────────────────────────┤
│   Service Layer (Business Logic) │  ← Interface + Implementation
├──────────────────────────────────┤
│   Repository Layer (Data Access) │  ← Spring Data JPA
├──────────────────────────────────┤
│   Entity Layer (Domain Model)    │  ← JPA Entities
└──────────────────────────────────┘
```

### Principles Followed

| Principle | Application |
|-----------|-------------|
| **SOLID** | Single‑responsibility services; open/closed via interfaces; Liskov‑safe impls |
| **DRY** | `ApiResponse<T>` wrapper reused across all endpoints |
| **Clean Code** | Meaningful names, small methods, no magic numbers |
| **Fail Fast** | Bean Validation (`@Valid`) + `GlobalExceptionHandler` |
| **TDD** | Tests are written **before** implementation code |
| **12‑Factor App** | Config via environment profiles; no hard‑coded secrets |

### Package Layout (per module)

```
module/<domain>/
├── controller/          # API interface (Swagger annotations live here)
├── controllerImpl/      # REST controller implementation
├── dto/                 # Request & Response DTOs (separate classes)
├── entity/              # JPA entity
├── repository/          # Spring Data JPA repository
├── service/             # Service interface
└── serviceImpl/         # Business logic implementation
```

## 5. Build & Run Instructions

### 5.1 Clone the Repository

```bash
git clone https://github.com/simple-makes-easy/be-patient-management-system.git
cd be-patient-management-system
```

### 5.2 Configure Local Database

Create a PostgreSQL database and user:

```sql
CREATE DATABASE patientdb;
CREATE USER postgres WITH PASSWORD 'root';
GRANT ALL PRIVILEGES ON DATABASE patientdb TO postgres;
```

Verify `src/main/resources/application-local.properties` contains the correct JDBC URL.

### 5.3 Build the Project

```bash
# Windows
.\gradlew.bat clean build

# macOS / Linux
./gradlew clean build
```

### 5.4 Run the Application (local profile)

```bash
.\gradlew.bat bootRun --args='--spring.profiles.active=local'
```

The server starts on **http://localhost:9080**

### 5.5 Run Only Unit Tests

```bash
.\gradlew.bat test --tests "*ServiceImplTest"
```

### 5.6 Run Only Integration Tests

> Docker must be running for Testcontainers to spin up a PostgreSQL container.

```bash
.\gradlew.bat test --tests "*IntegrationTest"
```

### 5.7 Run All Tests

```bash
.\gradlew.bat test
```

### 5.8 Build Executable JAR

```bash
.\gradlew.bat bootJar
java -jar build/libs/be-patient-management-system-0.0.1-SNAPSHOT.jar --spring.profiles.active=local
```

## 7. Database Migrations (Liquibase)

Migrations are managed by **Liquibase** and run automatically on application startup.

### Migration Folder Structure

```
db/changelogs/
├── db.changelog-master.xml          # Root: includes DDL then DML
├── ddl/
│   ├── changelog-ddl.xml            # Scans ddl/sql/ for all files
│   └── sql/
│       ├── *_01_create_trigger_function.sql
│       ├── *_02_create_patient_number_function.sql
│       ├── *_03_create_schema.sql
│       └── *_04_create_patient_trigger.sql
└── dml/
    ├── changelog-dml.xml            # Scans dml/sql/ for all files
    └── sql/
        └── *_insert_demo_data.sql
```

### Naming Convention for New SQL Files

```
DDMMYYYY[HHMMSS]_<sequence>_<description>.sql
```

Example: `02032026_001_add_patient_photo_url.sql`

Every SQL file must begin with Liquibase header comments:

```sql
--liquibase formatted sql
--changeset author:unique-changeset-id
--comment: Short description of what this changeset does
```

### Adding a New Migration

1. Create your `.sql` file under `ddl/sql/` (schema changes) or `dml/sql/` (data changes).
2. Follow the naming convention above.
3. The `<includeAll>` directive in the XML will pick it up automatically on next startup.

## 9. Testing Strategy (TDD)

This project follows **Test‑Driven Development**:

```
RED  → Write a failing test
GREEN → Write the minimum code to pass the test
REFACTOR → Clean up without breaking tests
```

### Test Layers

| Layer | Class Pattern | Tool | Scope |
|-------|--------------|------|-------|
| **Unit** | `*ServiceImplTest` | JUnit 5 + Mockito | Service logic in isolation (mocked repo) |
| **Integration** | `*IntegrationTest` | `@SpringBootTest` + Testcontainers | Full HTTP stack with real DB |

### Key Annotations

```java
// Unit test
@ExtendWith(MockitoExtension.class)
class PatientServiceImplTest { ... }

// Integration test
@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
@Import(TestcontainersConfiguration.class)
@ActiveProfiles("test")
class PatientControllerIntegrationTest { ... }
```

### Test Naming Convention

```java
@Test
void methodName_StateUnderTest_ExpectedBehavior() { ... }
// e.g.
void createPatient_ValidRequest_ReturnsCreatedPatient() { ... }
void getPatientById_NonExistentId_ThrowsResourceNotFoundException() { ... }
```

## 10. Coding Standards & Conventions

### General Rules

- Java 21 language features (`record`, `text block`, `sealed`, `var`) are encouraged.
- Use Lombok annotations to eliminate boilerplate (`@Getter`, `@Builder`, `@RequiredArgsConstructor`).
- No field injection (`@Autowired` on fields) — use **constructor injection only**.
- All public API methods must have `@Valid` on request DTOs.
- Service methods must **never** return JPA entities directly to controllers — always map to DTOs.
- Soft delete is preferred over hard delete where applicable.

### HTTP Status Code Contract

| Scenario | Status |
|----------|--------|
| Successful GET | `200 OK` |
| Successful create | `201 CREATED` |
| Successful update | `200 OK` |
| Successful delete | `204 NO CONTENT` |
| Validation failure | `400 BAD REQUEST` |
| Resource not found | `404 NOT FOUND` |
| Business rule violation | `409 CONFLICT` |
| Unexpected error | `500 INTERNAL SERVER ERROR` |

### DTO Conventions

- **Separate** request and response DTOs — never reuse the same class for both.
- Use `@Schema(description = "...", example = "...")` on every DTO field.
- Use `@NotBlank`, `@Email`, `@Size`, `@Past` etc. for validation.

### API Versioning

All endpoints are prefixed with `/api/v1/` (defined in `ApiConstants.java`).  
A new breaking version increments to `/api/v2/`.

## 11. Adding a New Module / Feature

1. **Create the test first** (TDD red phase):
   - Unit test: `src/test/.../module/<domain>/<Domain>ServiceImplTest.java`
   - Integration test: `src/test/.../module/<domain>/<Domain>ControllerIntegrationTest.java`

2. **Scaffold the module** under `src/main/.../module/<domain>/`:
   - `entity/` → JPA entity
   - `dto/` → `<Domain>RequestDto.java`, `<Domain>ResponseDto.java`
   - `repository/` → `<Domain>Repository.java`
   - `service/` → `<Domain>Service.java` (interface)
   - `serviceImpl/` → `<Domain>ServiceImpl.java`
   - `controller/` → `<Domain>Controller.java` (interface with Swagger)
   - `controllerImpl/` → `<Domain>ControllerImpl.java`

3. **Add a Liquibase migration** if a schema change is needed (see §7).

4. **Run tests — all must pass** before merging.
